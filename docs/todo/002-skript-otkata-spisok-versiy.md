# Скрипт отката на предыдущую версию и список версий в кластере

## Описание

Реализовать скрипт отката на предыдущую версию (+ список версий в кластере).

После реализации -- обновить документацию и демо (раздел `demo/`, в том числе
вспомогательные слайды) под новую операцию отката.

Дополнительно -- добавить типовые диаграммы развёртывания окружений dev и prod
(Mermaid -> `docs/pics/`). Для наших продуктов топология dev и prod идентична,

`KUBECONTEXT`/`REGISTRY`), а не две отдельные.

В конце задачи -- пересобрать оба PDF (документация без demo и демо со слайдами)
навыком `/pdf`, чтобы они вобрали операцию отката и новые типовые диаграммы
развёртывания.

### Разворот (rollback) werf

werf разворачивает приложение как helm-совместимый релиз, и история ревизий
релиза хранится в кластере (как у helm). Версия в этом контуре единая: `VERSION`
-> `CI_TAG`, образы с этим тегом уже лежат в registry после публикации. Отсюда
два опорных механизма отката, на которых строится скрипт:

- **Повторный `werf converge` на предыдущий `CI_TAG`/`VERSION`** -- образы
  целевой версии уже опубликованы, поэтому converge переразворачивает релиз без
  пересборки. Предпочтительный путь: проходит через тот же контракт
  `.helm/def.sh` и `utils/03-werf-converge.sh`, что и обычная публикация.
- **`helm rollback` к предыдущей ревизии релиза** -- быстрый откат манифестов к
  прошлой ревизии helm-релиза в неймспейсе `<NAMESPACE>-<ENVNAME>`, без участия
  registry.

**Список версий в кластере** -- источники сигналов: ревизии helm-релиза
(`helm history` по релизу в неймспейсе окружения) и/или опубликованные теги
образов в registry. Скрипт должен показывать доступные для отката версии перед
выбором цели.

Открытые вопросы для grill-me: какой механизм брать за основу (converge на тег
vs helm rollback); как соотносится откат с миграциями БД (init.sql/`predeploy`);
интеграция со существующими `kube_ci/<env>/`-скриптами (рядом с
`01-dissmiss.sh`); поведение при общем preprod-кластере dev/prod.

## Drill

Факты (сверено по коду): `converge` (`utils/03-werf-converge.sh`) собирает
образы из ТЕКУЩЕГО рабочего дерева и навешивает `--use-custom-tag=%image%-$CI_TAG`
как алиас; релиз -- helm в неймспейсе `<NAMESPACE>-<ENVNAME>`. `01-dissmiss.sh`
-- полный снос неймспейса (`werf dismiss --with-namespace`), не откат версии.
Версия: `VERSION` -> `CI_TAG` (`def.sh`); `set-version.sh bump` делает только
`git add`, без commit/tag (версии в git не тегируются). БД -- Postgres с
persistent-данными, инициализация `init.sql` через db-init ConfigMap.

Q1. Базовый механизм -> сначала выбран converge-на-CI_TAG, но вскрылся изъян:
converge собирает из текущего исходника, а не возвращает прошлый код (нужен
git-state прошлой версии, которого контур не тегирует).

Q3 (ре-решение). **helm rollback -- основа** (указание Архитектора: откат это
ЧП, нужен быстрый возврат). helm rollback восстанавливает сохранённую ревизию
релиза с прежней ссылкой на УЖЕ опубликованный образ -- без сборки и без
исходника, за секунды. converge-на-версию из ядра скрипта убран; остаётся
максимум абзацем в доке как «промоут конкретной версии» (требует её git-ref).

Q2. Список версий -> **`helm history`** по релизу в неймспейсе окружения
(revision -> appVersion/CI_TAG, дата, статус) как основа; перед откатом скрипт
сверяет, что образ целевой версии присутствует в registry.

Q4. БД -> **не трогать, явно предупредить**. Скрипт откатывает только релиз;
перед откатом печатает предупреждение, что схема БД не откатывается и при
forward-несовместимых миграциях нужен отдельный ручной шаг. В доке -- раздел
про это ограничение. (Дамп БД -- избыточно для демо/ЧП.)

Q5. Форма -> **отдельный env-скрипт** `kube_ci/<env>/03-rollback.sh`
(source productlist/k8s_defs + общий хелпер) + `utils/06-rollback.sh`
(helm history + helm rollback). Вызов: `03-rollback.sh <product> [revision]`.
Без revision -- печатает `helm history` и usage; с revision -- откат.
По-продуктно, явный product key (без `--all` по умолчанию, как у dismiss).

Дефолты (стир на скорость, не возражений):
- Общий preprod-кластер dev/prod -> откат работает по неймспейсу
  `<NAMESPACE>-<ENVNAME>`, окружения не пересекаются, спец-логики нет.
- Сопутствующее (из описания): типовая параметризованная диаграмма
  развёртывания dev/prod (Mermaid -> `docs/pics/`), правки документации
  (delivery/kube-ci-operations, runbooks/deploy) и демо (talk-plan/slides/
  qa-bank) под операцию отката, в конце -- пересборка обоих PDF навыком `/pdf`.

## Решения

- **Механизм отката** -- `helm rollback` к выбранной ревизии релиза в
  неймспейсе `<NAMESPACE>-<ENVNAME>`. Быстрый возврат ранее опубликованного
  образа без сборки и без исходника (откат -- ЧП-операция). converge-на-версию
  в ядро скрипта НЕ входит; в доке -- отдельный абзац как «промоут конкретной
  версии» (требует git-ref той версии).
- **Список версий** -- `helm history` по релизу в неймспейсе окружения
  (revision -> appVersion/CI_TAG, дата, статус). Перед откатом скрипт проверяет
  наличие образа целевой версии в registry.
- **БД** -- скрипт НЕ трогает БД. Перед откатом печатает предупреждение, что
  схема БД не откатывается (persistent Postgres, init.sql) и при
  forward-несовместимых миграциях нужен отдельный ручной шаг. В доке -- раздел
  про это ограничение.
- **Форма** -- отдельный env-скрипт `kube_ci/<env>/03-rollback.sh` (source
  `productlist`/`k8s_defs` + общий хелпер) и `utils/06-rollback.sh`
  (`helm history` + `helm rollback`). Вызов: `03-rollback.sh <product> [revision]`.
  Без revision -- печатает `helm history` и usage; с revision -- откат.
  По-продуктно, явный product key (без `--all` по умолчанию, как у dismiss).
- **Общий кластер** -- откат по неймспейсу окружения, dev/prod не пересекаются,
  спец-логики нет.
- **Сопутствующее** -- типовая параметризованная диаграмма развёртывания
  dev/prod (Mermaid -> `docs/pics/`); правки документации
  (`delivery/kube-ci-operations.md`, `runbooks/deploy.md`, при необходимости
  новый `runbooks/rollback.md`) и демо (`demo/talk-plan-40min.md`,
  `demo/slides.md`, `demo/qa-bank.md`) под операцию отката; в конце --
  пересборка обоих PDF навыком `/pdf` (через `scripts/createpdf/`, оба
  `doc-order-*`).
- **Согласованность контура** -- индексы README соответствующих разделов и
  путь чтения обновить под новые статьи/руководства; держать всё IP-чистым
  (без сырых `192.168.*` в публичных файлах -- требование git-guard).
