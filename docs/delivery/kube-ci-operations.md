# Операции kube_ci

Контур `kube_ci` сведён к четырём базовым операциям над уже развёрнутым
кластером: публикация (converge), откат версии (helm rollback), снос (dismiss) и
очистка (purge). Каждая запускается одной командой из каталога окружения
`kube_ci/dev/` или `kube_ci/prod/`; точки входа окружения лишь подключают общие
функции из [`kube_ci/utils/`](../../kube_ci/utils/03-werf-converge.sh) и
подмешивают локальные `k8s_defs` и `productlist`. Эта статья описывает, что
делает каждая операция и какие скрипты за ней стоят. Пошаговые сценарии не
дублируются -- они в [runbook деплоя](../runbooks/deploy.md) и
[runbook отката](../runbooks/rollback.md); модель окружений -- в
[dev-prod](dev-prod.md).

## Сводка

| Операция | Команда (из каталога окружения) | Библиотека |
|---|---|---|
| Публикация | `./pull_products.sh && ./00-build-deploy.sh [--all\|<product>]` | [`utils/03-werf-converge.sh`](../../kube_ci/utils/03-werf-converge.sh) |
| Откат версии | `./03-rollback.sh <product> [revision]` | [`utils/06-rollback.sh`](../../kube_ci/utils/06-rollback.sh) |
| Снос | `./01-dismiss.sh <product>\|--all` | [`utils/04-dismiss.sh`](../../kube_ci/utils/04-dismiss.sh) |
| Очистка | `./02-purge-stages.sh` | [`utils/05-purge-stages-local.sh`](../../kube_ci/utils/05-purge-stages-local.sh) |

Каждый файл `utils/*.sh` защищён от двойного выполнения проверкой
`[[ "${#BASH_SOURCE[@]}" -gt "1" ]] && return`, поэтому работает и как библиотека
(`source`), и как самостоятельный скрипт. Отличия этой копии оркестрации от
исходного `kube_ci` -- в [kube_ci/README.md](../../kube_ci/README.md).

## Публикация (converge)

Публикация состоит из двух шагов. Сначала
[`pull_products.sh`](../../kube_ci/dev/pull_products.sh) готовит продукты: создаёт
в каталоге окружения подкаталог `products/` и связывает в него каждый продукт из
`productlist` symlink'ом на `apps/<product>`. В демо исходники лежат в репозитории
рядом, поэтому вместо `git clone` -- symlink. В CI этот шаг заменяется
клонированием, см. [Подключение к GitLab CI](../integrations/gitlab-ci.md).

Затем [`00-build-deploy.sh`](../../kube_ci/dev/00-build-deploy.sh) выкатывает
продукты. Без аргументов или с `--all` он берёт все продукты из `productlist`,
иначе -- только перечисленные. Скрипт подключает библиотеку converge, читает
`productlist` и `k8s_defs`, экспортирует параметры кластера и для каждого
выбранного продукта заходит в его каталог и вызывает `deploy <env-функция>`.

Сам converge выполняет функция `deploy()` из
[`utils/03-werf-converge.sh`](../../kube_ci/utils/03-werf-converge.sh): активирует
werf через `trdl`, подключает контракт `.helm/def.sh`, вычисляет целевой
неймспейс `<NAMESPACE>-<ENVNAME>`, подхватывает `values-<ENVNAME>.yaml` и
`secrets-<ENVNAME>.yaml`, пробрасывает `CI_*`-переменные в helm и запускает `werf
converge` -- одной командой собирает образы, публикует их в реестр и применяет
релиз. Пошаговый разбор `deploy()` -- в
[Доставке в Kubernetes](../concepts/delivery-to-k8s.md). Тег образов несёт
единую версию продукта из файла `VERSION`, см. [Версионирование](versioning.md);
секреты окружения подмешиваются по ключу werf, см.
[Управление секретами](secrets.md).

Один запуск проходит так: инженер или CI вызывает `pull_products` и
`00-build-deploy`, контур читает контракт и передаёт управление werf, werf
собирает образы, кладёт их в registry, рендерит чарт и применяет манифесты через
Kubernetes API, после чего postdeploy печатает URL ресурсов.

## Откат версии (helm rollback)

[`03-rollback.sh`](../../kube_ci/dev/03-rollback.sh) возвращает релиз продукта на
ранее опубликованную ревизию. Под капотом
[`utils/06-rollback.sh`](../../kube_ci/utils/06-rollback.sh) подключает контракт,
вызывает env-функцию, вычисляет имя релиза `${APPNAME}-${ENVNAME}` и неймспейс
`<NAMESPACE>-<ENVNAME>`, после чего работает с историей релиза через `helm`.

Это операция на случай чрезвычайной ситуации: образ целевой ревизии уже лежит в
registry, поэтому возврат проходит за секунды -- без пересборки и без исходного
кода целевой версии. Откат не переразворачивает образы из рабочего дерева, а
переключает релиз на сохранённую ревизию helm.

Скрипт требует явного ключа продукта -- по умолчанию он не работает по `--all`,
как и снос. Без аргумента `03-rollback.sh` завершается с ошибкой и печатает
список доступных продуктов. Дальше поведение зависит от второго аргумента:

- `<product>` без `revision` -- скрипт печатает `helm history` релиза в
  неймспейсе окружения: каждая строка -- ревизия с датой, статусом и `APP
  VERSION` (версия продукта, она же `CI_TAG` ревизии). Это список версий, из
  которого выбирают цель отката;
- `<product> <revision>` -- скрипт печатает предупреждение про БД, проверяет
  наличие тегов образа продукта в registry через HTTP API v2
  (`/v2/<APPNAME>/tags/list`) и выполняет `helm rollback <release> <revision>`.

### Список версий через helm history

Доступные для отката версии хранит сам helm: каждая публикация через converge
создаёт новую ревизию релиза. `helm history` по релизу `${APPNAME}-${ENVNAME}` в
неймспейсе `<NAMESPACE>-<ENVNAME>` даёт сопоставление ревизия -> `APP VERSION`
(`CI_TAG`), дата, статус. Контур не тегирует версии в git (см.
[Версионирование](versioning.md): `VERSION` -> `CI_TAG`, `set-version.sh` делает
только `git add`), поэтому история helm-релиза -- единственный реестр того, какие
версии продукта уже выкатывались в это окружение.

### Ограничение: БД не откатывается

`helm rollback` возвращает только релиз продукта -- Deployment-ы, Service-ы,
ConfigMap-ы, Secret-ы. Схема и данные PostgreSQL остаются как есть: Postgres
держит данные на persistent-томе, а `init.sql` применяется только при первой
инициализации пустого тома. Если между текущей и целевой версией была
forward-несовместимая миграция схемы, откат релиза вернёт старый код поверх новой
схемы. Приведение схемы к целевой версии -- отдельный ручной шаг, в ядро отката
он не входит; скрипт печатает об этом предупреждение перед `helm rollback`.

### Откат против промоута конкретной версии

Откат через `helm rollback` -- не единственный способ оказаться на нужной версии,
но единственный пригодный для чрезвычайной ситуации. Промоут конкретной версии
повторным converge тоже возможен, но требует git-ref той версии в рабочем дереве:
converge переразворачивает образы из текущего состояния репозитория, а не из
сохранённой ревизии. Контур версии в git не тегирует, поэтому быстро встать на
произвольную прошлую версию через converge нельзя -- для этого пришлось бы вручную
восстановить рабочее дерево той версии и пересобрать образы. Для быстрого возврата
после неудачной выкатки используют `03-rollback.sh`.

## Снос (dismiss)

[`01-dismiss.sh`](../../kube_ci/dev/01-dismiss.sh) снимает релиз целиком. Под
капотом [`utils/04-dismiss.sh`](../../kube_ci/utils/04-dismiss.sh) подключает
контракт, вызывает env-функцию и выполняет `werf dismiss --with-namespace` --
сносит релиз вместе с его неймспейсом. Это не откат версии: снос удаляет
развёртывание, а не переключает его на прошлую ревизию.

Скрипт требует явного аргумента: один или несколько ключей продуктов либо
`--all`. Без аргумента он отказывается работать и завершается с ошибкой -- иначе
обход `productlist` снял бы все продукты разом. Если переданный аргумент не
совпал ни с одним продуктом, скрипт тоже завершается с ошибкой и печатает список
доступных ключей. Эта защита особенно важна для prod, где снос -- точка
остановки (см. [dev-prod](dev-prod.md)).

## Очистка (purge)

[`02-purge-stages.sh`](../../kube_ci/dev/02-purge-stages.sh) сбрасывает локальный
кеш стадий сборки werf. Через
[`utils/05-purge-stages-local.sh`](../../kube_ci/utils/05-purge-stages-local.sh)
он вызывает `werf stages purge --stages-storage=:local` по каждому продукту из
`productlist`. Это освобождает место, а следующая сборка идёт с нуля. Очистка не
трогает развёрнутые релизы -- это операция над кешем сборщика, а не над
кластером.

Более жёсткий вариант -- [`utils/10-purge-werf-registry.sh`](../../kube_ci/utils/10-purge-werf-registry.sh):
он запускает `werf host cleanup` и `werf host purge` и удаляет docker-образы werf
по шаблонам `werf-images`, `werf-stages`, `werf-managed`. Этот скрипт не
привязан к окружению и работает по всему хосту сборки.

## Типовая топология окружения

Топология dev и prod идентична -- различаются только параметры окружения:
`ENVNAME`, неймспейс `<NAMESPACE>-<ENVNAME>`, хост ingress в `CI_URL`,
`KUBECONTEXT` и `REGISTRY` из `k8s_defs`. Схема ниже показывает путь от рабочей
машины (только `git push`) через машину деплоя (`git pull` плюс прогон `kube_ci`)
к кластеру окружения: converge выкатывает релиз и пушит образы в registry, откат
переключает релиз на прошлую ревизию, а внутри кластера ingress ведёт на сервисы
продукта, Postgres и pgAdmin.

## Плюсы, минусы, безопасность

Плюсы. Весь жизненный цикл доставки сведён к небольшому набору команд с
предсказуемым поведением; логика вынесена в общие библиотеки и не дублируется
между окружениями. Откат отделён от сноса: неудачную выкатку возвращают за
секунды через `helm rollback`, не пересобирая образы.

Минусы. Снос и очистка `dismiss`/`purge` -- разрушающие операции; защита от
случайного запуска есть только у сноса (обязательный аргумент), очистка кеша
выполняется без подтверждения. Откат версии не трогает БД, поэтому при
несовместимой миграции схему приводят к целевой версии вручную.

Безопасность. Операции prod -- точка остановки, запускаются только с разрешения
оператора; общий для dev и prod кластер усиливает цену ошибки. Разбор рисков --
в [Компромиссах и безопасности схемы](../concepts/security-and-tradeoffs.md).
