# Runbook: откат версии релиза в dev / prod

Возврат релиза продукта на ранее опубликованную ревизию через `helm rollback`.
Операция на случай чрезвычайной ситуации: после неудачной выкатки она за секунды
возвращает релиз к рабочей версии -- образ целевой ревизии уже в registry, поэтому
ни пересборки, ни исходного кода целевой версии не требуется. Команда одна,
меняется только каталог окружения (`kube_ci/dev`, `kube_ci/prod`). Ниже -- на
примере `dev`.

ВРЕМЕННО оба окружения указывают на один физический preprod-кластер; продукты
различаются неймспейсом (`<NAMESPACE>-dev`, `-prod`). Откат версии -- обратная к
публикации операция; снос релиза (`01-dissmiss.sh`) -- другая операция и описан в
[runbook деплоя](deploy.md).

## Предусловия

- Продукт уже публиковался в это окружение хотя бы один раз -- у релиза
  `<APPNAME>-<ENVNAME>` есть история ревизий. Публикация -- в
  [runbook деплоя](deploy.md).
- Кластер окружения доступен, в `~/.kube/config` есть его контекст -- см.
  [cluster-connection.md](cluster-connection.md).
- `werf` подтягивается через `trdl` (`trdl use werf 2 stable`); скрипт берёт
  совместимый `helm` из поставки werf.
- Запуск из каталога окружения (`kube_ci/dev/` или `kube_ci/prod/`). Операции
  prod -- точка остановки: их запускают только с явного разрешения оператора.

## Шаг 1. Посмотреть доступные версии

Без второго аргумента скрипт печатает историю ревизий релиза и не меняет
кластер:

```bash
cd kube_ci/dev
./03-rollback.sh app1-java-react
```

Скрипт вызывает `helm history` по релизу `<APPNAME>-<ENVNAME>` в неймспейсе
`<NAMESPACE>-<ENVNAME>`. Каждая строка -- ревизия helm: номер `REVISION`, дата
обновления, статус (`deployed`, `superseded`, `failed`), `CHART`, `APP VERSION`.
Колонка `APP VERSION` -- это версия продукта (`CI_TAG`), которая выкатывалась
этой ревизией. История helm-релиза -- единственный реестр уже выкаченных версий:
контур не тегирует версии в git (см.
[Версионирование](../delivery/versioning.md)).

Цель отката выбирают по этой таблице -- как правило, последнюю ревизию со
статусом `deployed`, предшествующую неудачной выкатке.

## Шаг 2. Выбрать ревизию и откатить

```bash
cd kube_ci/dev
./03-rollback.sh app1-java-react 3   # откат на ревизию 3
```

Перед `helm rollback` скрипт:

- печатает предупреждение про БД (см. ниже);
- проверяет наличие тегов образа продукта в registry через HTTP API v2
  (`/v2/<APPNAME>/tags/list`). Если registry не ответил, скрипт печатает
  предупреждение и продолжает: ссылка на образ берётся из манифестов целевой
  ревизии helm;
- выполняет `helm rollback <APPNAME>-<ENVNAME> <revision>` в неймспейсе
  окружения.

helm создаёт новую ревизию, повторяющую состояние целевой, и применяет её
манифесты. Релиз возвращается к образам и конфигурации выбранной версии.

## Предупреждение: БД не откатывается

`helm rollback` возвращает только релиз продукта -- Deployment-ы, Service-ы,
ConfigMap-ы, Secret-ы. Схема и данные PostgreSQL остаются как есть: Postgres
держит данные на persistent-томе, а `init.sql` применяется только при первой
инициализации пустого тома. Если между текущей и целевой версией прошла
forward-несовместимая миграция, откат вернёт старый код продукта поверх новой
схемы. Приведение схемы к целевой версии -- отдельный ручной шаг; в ядро отката
он не входит. Перед откатом оцените, менялась ли схема между версиями.

## Шаг 3. Проверить результат

```bash
./03-rollback.sh app1-java-react              # верхняя строка history -- новая deployed-ревизия
kubectl -n app1-java-react-dev get pods       # поды пересозданы и в Running
kubectl -n app1-java-react-dev get deploy -o wide   # образы несут тег целевой версии
```

Откройте фронт на хосте `<app>-dev-<NODE_IP>.nip.io`: бекенд должен отдавать
версию целевой ревизии. Если рабочая нагрузка читает данные несовместимой схемы --
вернитесь к разделу про БД.

## Troubleshooting

- Релиз не найден / пустая история -- продукт в это окружение не публиковался,
  откатывать нечего; сперва опубликуйте (см. [runbook деплоя](deploy.md)).
- `helm rollback` падает на применении манифестов, ImagePullBackOff, таймаут,
  отсутствие контекста -- разбор типовых сбоев в
  [troubleshooting.md](troubleshooting.md).
- Список тегов образа недоступен -- registry HTTP API не ответил; проверьте
  доступность и insecure-настройку реестра в
  [cluster-connection.md](cluster-connection.md).
