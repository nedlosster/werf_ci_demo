# Введение в werf

[werf](https://werf.io) -- open-source CLI-инструмент сборки образов и доставки
приложений в Kubernetes. В демо werf -- единственный инструмент, который и
собирает образы, и разворачивает релиз; bash-обёртки kube_ci лишь вызывают его
с нужными параметрами.

## Что делает werf

| Возможность | Кратко |
|---|---|
| Сборка образов | по `werf.yaml`; кеширование по стадиям (stages) |
| Публикация | пуш образов в container registry |
| Деплой | helm-совместимый чарт `.helm/`, релиз в Kubernetes |
| Очистка | сборка мусора в registry по политикам (`cleanup`) |

Ключевая команда -- `werf converge`: одной операцией собирает недостающие
образы, публикует их в registry и применяет релиз в кластер (по сути
`build + publish + deploy`). Именно её вызывает `utils/03-werf-converge.sh`.

## Giterminism

werf по умолчанию требует, чтобы конфигурация была закоммичена в git
(воспроизводимость сборки). В демо продукты подтягиваются локально и могут
собираться из рабочего дерева, поэтому используется послабление
`--loose-giterminism=true` (и `--dev` для локальной разработки).

## Stages и stage storage

Сборка разбита на стадии; собранные стадии кешируются. В демо используется
локальное хранилище стадий (`--synchronization :local`). Сброс кеша --
`werf stages purge` (см. `utils/05-purge-stages-local.sh`).

## Чарт `.helm/`

werf использует helm-совместимый чарт продукта (`Chart.yaml`, `templates/`,
`values-<env>.yaml`, секреты `secrets-<env>.yaml`). Значения окружения
прокидываются через `--set` и `--values` (контракт продукта --
[../apps/README.md](../apps/README.md)).

## Установка через trdl

werf ставится менеджером версий [trdl](https://trdl.dev) и активируется в
сессии:

```bash
source "$(~/bin/trdl use werf 2 stable)"
```

В демо все скрипты используют единый канал `werf 2 stable` (старый `multiwerf`
не используется). Точные команды установки trdl и добавления репозитория werf
(включая TUF root-key -- security-релевантное значение) брать на официальной
странице https://werf.io/getting_started/.
