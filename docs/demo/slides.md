# Слайды доклада

Аутлайн вспомогательных слайдов для доклада на 60 минут. Один слайд -- заголовок,
привязанный к блоку из [плана доклада](talk-plan-60min.md), и три-пять коротких
тезисов в телеграфном стиле, не абзацы прозы. Строка «Показать» указывает на
диаграмму или статью, которую держат на экране в этом блоке.

Как пользоваться: слайды идут по блокам доклада 1-10; нумерация слайдов сквозная.
Блок 11 «Вопросы» отдельного контент-слайда не требует -- для него служат
backup-слайды из [банка вопросов](qa-bank.md), которые держат скрытыми и
показывают по запросу. Тезисы здесь -- опорные точки, а не текст для зачитывания;
развёрнутая аргументация и тайминг -- в [плане доклада](talk-plan-60min.md).

Колода ранжирована под ядро часа: контракт и развязка, разработка прямо в
кластере с живым циклом на app1, операции на живых командах. Блоки про
компромиссы, CI, DORA, секреты и версии собраны в один контекстный слайд.

Диаграммы лежат в [pics/](../pics/): `architecture`, `converge-flow`,
`industrial-cicd`, `product-anatomy`, `dora-flow`, `deploy-topology`,
`dev-workflow-cycle`.

## Слайд 1. Титул

- демо: единый CI-контур ставит разнородные приложения в готовые кластеры dev и prod
- дистилляция личного опыта разработки под Kubernetes за 2020-2025, не учебный пример
- граница доклада -- как доставляем и как разрабатываем, а не что доставляем
- два контрастных продукта, общий контракт `.helm/def.sh`, разработка прямо в кластере

Показать: титульный кадр и [индекс документации](../README.md) одним экраном.

## Слайд 2. Дистилляция опыта 2020-2025 (блок 1)

- контур и приёмы выросли из реальной работы, а не из учебного примера
- демонстрируется способ доставки и разработки, не приложение
- кластеры считаем уже развёрнутыми, их разворачивание вне скопа
- бизнес-логика продуктов минимальна по замыслу

Показать: [обзор продуктов](../products/overview.md).

## Слайд 3. Проблема доставки в Kubernetes (блок 2)

- доставка распадается на сборку образа и применение релиза
- большинство инструментов закрывают одну половину
- стык держит инженер: синхронизация тегов, пересборка слоёв
- работа повторяется в каждой команде и плохо переносится

Показать: [доставка в Kubernetes](../concepts/delivery-to-k8s.md), схема industrial-cicd ([industrial-cicd.png](../pics/industrial-cicd.png)).

## Слайд 4. werf converge сшивает обе половины (блок 2)

- сборка и применение релиза в одной программе из одного git-состояния
- `werf converge`: собрать, опубликовать, применить; идемпотентен, в демо атомарен
- сборка по кешируемым стадиям, пересборка только затронутого
- giterminism по умолчанию; в демо ослаблен -- см. контекстный слайд

Показать: [введение в werf](../concepts/werf-intro.md), [поток converge](../pics/converge-flow.png).

## Слайд 5. Контур kube_ci и контракт (блок 3, ядро 1)

- тонкая bash-обёртка над werf, не своя система сборки
- базовые операции над готовым кластером; логика -- в `utils/`
- окружения подмешивают локальные `k8s_defs` и `productlist`
- контракт `.helm/def.sh`: `APPNAME`, `ENVNAME`, `CI_URL`, `CI_TAG`, опц. `NAMESPACE`
- `deploy()` читает только `${!CI_@}` и пробрасывает в helm; контур не знает продукт

Показать: [диаграмма архитектуры](../pics/architecture.png), [контракт приложения](../concepts/application-contract.md).

## Слайд 6. Два контрастных продукта (блок 4, ядро 1)

- `app1-java-react`: React + Spring Boot
- `app2-python-angular`: Angular + FastAPI
- разведены по языку, веб-фреймворку, менеджерам пакетов (Maven/pip, pnpm/npm), health-пробам
- один converge на оба, без ветки «если это app1»
- различия живут в `Dockerfile`, `werf.yaml`, чарте

Показать: [обзор продуктов](../products/overview.md), таблица стеков, [анатомия продукта](../pics/product-anatomy.png).

## Слайд 7. Разработка прямо в кластере (блок 5, ядро 2)

- dev -- долгоживущий под-песочница (`sleep infinity`) на StatefulSet, VS Code Remote
- dev собран из тех же кубовых объектов, что prod -- расхождение сред почти исчезает
- тяжёлое (maven, node, pip, базы) на ноде; редактор лёгкий, ноутбук не нагружен
- один артефакт доставки: dev и prod различаются только окружением в `.helm/def.sh`

Показать: [разработка внутри кластера](../delivery/dev-in-cluster.md), `kubectl -n app1-java-react-dev get pods`.

## Слайд 8. dev в кластере vs docker-compose (блок 5, ядро 2)

- docker-compose моделирует только контейнеры и docker-сеть -- Kubernetes-слоя нет
- в кластере отлаживается то, чего compose не показывает:
- Ingress-маршрутизация (`/api`, `/pgadmin`, host nip.io, rewrite), service discovery по DNS и `targetPort`
- readiness/liveness-пробы, rollout, `updateStrategy: OnDelete` у Postgres
- монтирование Secret и ConfigMap, init-контейнеры, сам `werf converge` с хуками

Показать: [разработка внутри кластера](../delivery/dev-in-cluster.md), раздел «Зачем dev в кластере, а не docker-compose».

## Слайд 9. Живой цикл: две зоны (блок 6, ядро 2, пик)

- внутренняя петля правка -> отладка -> тест замыкается на dev-поде, повторяется часто
- внешний контур push -> выкат через `kube_ci` проходится реже
- синхронизация только через git: ни `scp`, ни `rsync` между машинами
- dev-сервер фронта app1 -- `dev-start.sh`: `npm install` + `vite --host 0.0.0.0 --port 8080`

Показать: диаграмма [цикла работы в dev-поде](../pics/dev-workflow-cycle.png).

## Слайд 10. Демонстрация на app1 (блок 6, ядро 2, пик)

- правка React-компонента -> Vite HMR мгновенно в браузере через ingress (nip.io)
- сервер виден снаружи: слушает `targetPort` dev-Service, `allowedHosts` `.nip.io` в `vite.config.ts`
- проверено в dev -> `git push` с рабочей машины -> converge в dev с новой версией
- сборку прогнать заранее/параллельно; запасной вариант -- заранее снятый результат

Показать: живая правка фронта app1 на `app1-java-react-dev-<NODE_IP>.nip.io`, вывод converge с новой версией.

## Слайд 11. Кеши и persistent-volume'ы dev-подов (блок 7)

- два PVC на под: `workspace` (клон, исходники, `node_modules`, артефакты) и `homeapp` (кеши, `.vscode-server`)
- симлинки `~/.m2`, `~/.npm`, `~/.cache`, `~/.local` ведут на том `homeapp`
- первый клон и `npm install` -- разовая плата; дальше из кеша без сети
- тома живут дольше пода: рестарт и перезагрузка ВМ их не трогают

Показать: [кеши и persistent-volume'ы dev-сред](../delivery/dev-caches-and-volumes.md), таблица двух PVC.

## Слайд 12. Операции kube_ci (блок 8, ядро 3)

- публикация: `./pull_products.sh && ./00-build-deploy.sh [--all|<product>]`
- откат версии: `./03-rollback.sh <product> [revision]`, `helm rollback`, БД не трогает
- снос: `./01-dismiss.sh <product>|--all`, без аргумента отказывается
- очистка: `./02-purge-stages.sh`, сбрасывает кеш стадий, релизы не трогает

Показать: топология окружения ([deploy-topology.png](../pics/deploy-topology.png)), живой прогон публикации в dev.

## Слайд 13. dev и prod одним контуром (блок 8, ядро 3)

- `kube_ci/dev/` и `kube_ci/prod/` -- самодостаточные каталоги, одинаковый набор файлов
- различие -- `k8s_defs` и env-функция продукта из `productlist`
- релиз в неймспейс `<NAMESPACE>-<ENVNAME>`, dev и prod не конфликтуют
- временно один preprod-кластер; разделение -- смена `KUBECONTEXT` и `REGISTRY`

Показать: [один контур, два окружения](../delivery/dev-prod.md), вывод `kubectl get ns`.

## Слайд 14. Контекст: компромиссы, CI, DORA, секреты, версии (блок 9)

- послабления обратимы: insecure-registry, `--loose-giterminism`, общий кластер dev/prod, nip.io, ключ werf, dev-SSH-под
- те же скрипты в пайплайне CI: `deploy-dev`/`deploy-prod`, `rollback`, `cleanup`; автодеплой dev, ручной промоут prod
- четыре метрики DORA с артефактов контура; кода сбора в репозитории нет
- секреты в `.helm/secrets-<env>.yaml` шифрует werf по `WERF_SECRET_KEY`; версия `VERSION` -> `CI_TAG`

Показать: [компромиссы и безопасность схемы](../concepts/security-and-tradeoffs.md) одним кадром-перечнем.

## Слайд 15. Выводы (блок 10)

- контракт `.helm/def.sh` -- два разнородных продукта одним кодом
- разработка прямо в кластере: dev≈prod, отладка k8s, цикл правка -> push -> converge через git
- werf сшивает сборку и релиз из одного git-состояния, доставка -- набор операций
- послабления демо понятны и обратимы: в проде меняется конфигурация, не логика
- всё это -- дистилляция практики 2020-2025

Показать: [диаграмма архитектуры](../pics/architecture.png) финальным кадром.
