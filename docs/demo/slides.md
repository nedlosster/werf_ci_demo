# Слайды доклада

Аутлайн вспомогательных слайдов для доклада на 40 минут. Один слайд -- заголовок,
привязанный к блоку из [плана доклада](talk-plan-40min.md), и три-пять коротких
тезисов в телеграфном стиле, не абзацы прозы. Строка «Показать» указывает на
диаграмму или статью, которую держат на экране в этом блоке.

Как пользоваться: слайды идут по блокам доклада 1-11; нумерация слайдов сквозная.
Блок 12 «Вопросы» отдельного контент-слайда не требует -- для него служат
backup-слайды из [банка вопросов](qa-bank.md), которые держат скрытыми и
показывают по запросу. Тезисы здесь -- опорные точки, а не текст для зачитывания;
развёрнутая аргументация и тайминг -- в [плане доклада](talk-plan-40min.md).

Диаграммы лежат в [pics/](../pics/): `architecture`, `converge-flow`,
`industrial-cicd`, `product-anatomy`, `dora-flow`, `deploy-topology`.

## Слайд 1. Титул

- демо: единый CI-контур ставит разнородные приложения в готовые кластеры dev и prod
- граница доклада -- как доставляем, а не что доставляем
- два контрастных продукта, общий контракт `.helm/def.sh`, базовые операции доставки

Показать: титульный кадр и [индекс документации](../README.md) одним экраном.

## Слайд 2. Цель демо (блок 1)

- демонстрируется способ доставки, не приложение
- кластеры считаем уже развёрнутыми, их разворачивание вне скопа
- бизнес-логика продуктов минимальна по замыслу

Показать: [обзор продуктов](../products/overview.md).

## Слайд 3. Проблема доставки в Kubernetes (блок 2)

- доставка распадается на сборку образа и применение релиза
- большинство инструментов закрывают одну половину
- стык держит инженер: синхронизация тегов, пересборка слоёв
- работа повторяется в каждой команде и плохо переносится

Показать: [доставка в Kubernetes](../concepts/delivery-to-k8s.md), схема industrial-cicd ([industrial-cicd.png](../pics/industrial-cicd.png)).

## Слайд 4. Что такое werf (блок 3)

- сборка и применение релиза в одной программе из одного git-состояния
- центральная команда -- `werf converge`: собрать, опубликовать, применить
- сборка по кешируемым стадиям, пересборка только затронутого
- giterminism по умолчанию; в демо ослаблен -- см. блок компромиссов

Показать: [введение в werf](../concepts/werf-intro.md), раздел про converge.

## Слайд 5. Поток converge (блок 3)

- converge идемпотентен и в демо запускается атомарно
- участники: инженер или CI -> kube_ci -> werf -> registry -> Kubernetes API
- werf сам решает, какие образы досбирать перед релизом

Показать: [поток converge](../pics/converge-flow.png).

## Слайд 6. Контур kube_ci (блок 4)

- тонкая bash-обёртка над werf, не своя система сборки
- базовые операции над готовым кластером; логика -- в `utils/`
- окружения подмешивают локальные `k8s_defs` и `productlist`
- контракт `.helm/def.sh`: `APPNAME`, `ENVNAME`, `CI_URL`, `CI_TAG`, опц. `NAMESPACE`

Показать: [диаграмма архитектуры](../pics/architecture.png), [контракт продукта](../../apps/README.md).

## Слайд 7. Два контрастных продукта (блок 5)

- `app1-java-react`: React + Spring Boot
- `app2-python-angular`: Angular + FastAPI
- разведены по языку, веб-фреймворку, менеджерам пакетов (Maven/pip, pnpm/npm), health-пробам
- один converge на оба, без ветки «если это app1»
- различия живут в `Dockerfile`, `werf.yaml`, чарте

Показать: [обзор продуктов](../products/overview.md), таблица сопоставления стеков.

## Слайд 8. Анатомия продукта (блок 5)

- неймспейс `<app>-<env>`: Deployment-ы фронта и бэкенда
- PostgreSQL, pgAdmin, Service-ы, Ingress
- Secret и ConfigMap db-init на старте

Показать: [анатомия продукта](../pics/product-anatomy.png).

## Слайд 9. Dev и prod одним контуром (блок 6)

- `kube_ci/dev/` и `kube_ci/prod/` -- самодостаточные каталоги, одинаковый набор файлов
- различие -- `k8s_defs` и env-функция продукта из `productlist`
- релиз в неймспейс `<NAMESPACE>-<ENVNAME>`, dev и prod не конфликтуют
- временно один preprod-кластер; разделение -- смена `KUBECONTEXT` и `REGISTRY`

Показать: [один контур, два окружения](../delivery/dev-prod.md), вывод `kubectl get ns`.

## Слайд 10. Операции kube_ci (блок 7)

- публикация: `./pull_products.sh && ./00-build-deploy.sh [--all|<product>]`
- откат версии: `./03-rollback.sh <product> [revision]`, `helm rollback`
- снос: `./01-dismiss.sh <product>|--all`, без аргумента отказывается
- очистка: `./02-purge-stages.sh`, сбрасывает кеш стадий, релизы не трогает
- снос и очистка разрушающие; топология окружения одна на dev и prod

Показать: топология окружения ([deploy-topology.png](../pics/deploy-topology.png)),
[runbook деплоя](../runbooks/deploy.md), живой прогон публикации в dev.

## Слайд 10а. Откат версии (блок 7)

- ЧП-операция: возврат релиза на ранее опубликованную ревизию за секунды
- образ целевой ревизии уже в registry -- без пересборки и исходников версии
- `03-rollback.sh <product>` без revision -- `helm history`, список выкаченных версий
- `helm rollback`, а не повторный converge: converge собирает из рабочего дерева
- БД не откатывается: схема Postgres persistent, миграцию схемы правят вручную

Показать: [runbook отката](../runbooks/rollback.md), живой `03-rollback.sh <product>`
с историей ревизий.

## Слайд 11. Секреты (блок 8)

- пароли в `.helm/secrets-<env>.yaml`, зашифрованы, коммитятся
- шифрует и расшифровывает werf по `WERF_SECRET_KEY`, ключ в репозиторий не попадает
- при converge `deploy()` добавляет `--secret-values`, werf подставляет значения в чарт

Показать: [управление секретами](../delivery/secrets.md), таблица раскладки файлов.

## Слайд 12. Версионирование (блок 8)

- источник истины -- файл `VERSION`, единая версия на продукт
- `set-version.sh` раскатывает версию по бэкенду, фронту, чарту
- `.helm/def.sh` отдаёт ту же версию как `CI_TAG` -- тег образов при выкатке

Показать: [версионирование](../delivery/versioning.md), схема `VERSION -> CI_TAG`.

## Слайд 13. Выход во внешний CI (блок 9)

- пайплайн -- тонкая обёртка над тремя скриптами: `deploy-dev`/`deploy-prod`, `rollback`, `cleanup`
- типовой поток: автодеплой в dev, ручной промоут в prod
- ключ werf и kubeconfig -- masked/protected-переменные CI

Показать: [подключение к GitLab CI](../integrations/gitlab-ci.md), пример `.gitlab-ci.yml`.

## Слайд 14. Метрики DORA (блок 9)

- четыре метрики снимаются с артефактов контура
- точки съёма: статусы и метки времени job-ов, git-история с `VERSION`, факты запуска скриптов
- кода сбора в репозитории нет: описана концепция и точки съёма

Показать: [поток DORA-метрик](../pics/dora-flow.png), [метрики DORA](../integrations/dora-metrics.md).

## Слайд 15. Компромиссы схемы (блок 10)

- registry в insecure-режиме (HTTP / skip-TLS) ради in-cluster registry на nip.io
- converge с `--loose-giterminism`: из рабочего дерева, теряется воспроизводимость
- dev и prod делят один кластер, изоляция только неймспейсом
- nip.io вместо реального DNS и валидного TLS
- риск секретов сосредоточен в `WERF_SECRET_KEY` на машине сборки
- dev-под клонирует монорепо по ssh-ключу оператора из `.helm/tmp/`

Показать: [компромиссы и безопасность схемы](../concepts/security-and-tradeoffs.md).

## Слайд 16. Выводы (блок 11)

- контракт `.helm/def.sh` -- два разнородных продукта одним кодом
- werf сшивает сборку и релиз из одного git-состояния, доставка -- набор операций
- послабления демо понятны и обратимы: в проде меняется конфигурация, не логика

Показать: [диаграмма архитектуры](../pics/architecture.png) финальным кадром.
