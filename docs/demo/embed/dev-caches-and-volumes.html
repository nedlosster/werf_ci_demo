Кеши и persistent-volume’ы dev-сред

dev-под – долгоживущая песочница (sleep infinity), к которой разработчик подключается через VS Code Remote. При каждом пересоздании пода сборка, установка зависимостей и клон монорепо запускались бы заново, поэтому их результаты вынесены на persistent-volume’ы. Статья разбирает, какие тома объявляет dev-StatefulSet, как симлинки из домашнего каталога перенаправляют кеши пакетных менеджеров на PVC и что переживает рестарт пода, перезагрузку ВМ и смену версии образа. Саму dev-схему (что рендерит окружение, подключение, запуск dev-серверов) описывает Разработка внутри кластера.

Зачем выносить состояние на тома

Контейнер dev-пода не запускает приложение – он спит, а сборку и dev-серверы разработчик поднимает вручную внутри пода. Полезное состояние такой песочницы (склонированный исходник, скачанные зависимости, артефакты сборки) накапливается в runtime и в образ не входит. Без отдельного хранилища оно пропадало бы при каждом пересоздании пода: converge с обновлённым шаблоном, bump версии образа, перепланировка пода на другую ноду, перезагрузка ВМ. Восстановление означало бы повторный клон репозитория и повторную закачку всех пакетов из сети.

Отсюда два PVC на каждый под. Первый клон и первая установка зависимостей выполняются единожды; дальше под собирается из локального кеша без обращения к сети.

Два PVC на каждый под

dev-StatefulSet объявляет volumeClaimTemplates с двумя томами. На каждый свой под StatefulSet создаёт отдельную пару PVC, поэтому тома не общие: у каждого из четырёх dev-подов своя пара. Шаблон – например, 031-backend-dev.yaml для бекенда app1; у трёх остальных компонентов он структурно идентичен.

Том mountPath Размер Содержимое
workspace /workspace 5Gi клон монорепо (/workspace/werf_ci_demo), исходники, node_modules компонентов, артефакты сборки (target/, dist/, .angular/)
homeapp /home/app/homeapp 5Gi каталоги кешей пакетных менеджеров, подключённые симлинками из ~/

Размеры и storageClassName берутся из values чарта (dev.workspaceSize, dev.homeappSize, dev.storageClass; по умолчанию 5Gi, 5Gi, standard). Оба тома монтируются и в init-контейнер prepare-dev-env, и в основной контейнер.

Имена PVC StatefulSet формирует как <том>-<под>, например workspace-app1-java-react-backend-dev-0 и homeapp-app1-java-react-backend-dev-0. На четыре dev-пода (два компонента в каждом из двух продуктов) приходится восемь PVC. Физически тома лежат на диске ноды через storageClass standard, но это раздельные claim’ы с раздельными каталогами – не общий каталог.

Распределение содержимого по томам:

Под workspace homeapp
app1 backend клон, исходники, target/ ~/.m2 (maven)
app1 frontend клон, исходники, node_modules, dist/ ~/.npm, ~/.cache
app2 backend клон, исходники ~/.cache (pip), ~/.local (pip install --user)
app2 frontend клон, исходники, node_modules, .angular/ ~/.npm, ~/.cache

Механизм симлинков

Пакетные менеджеры пишут кеши в фиксированные пути под домашним каталогом (~/.m2, ~/.npm, ~/.cache, ~/.local). На том homeapp эти пути перенаправляются в два шага.

  1. На этапе сборки образа Dockerfile.dev заменяет каждый такой каталог симлинком на соответствующий подкаталог /home/app/homeapp – например ln -sfn /home/app/homeapp/.m2 /home/app/.m2. На момент сборки целевой каталог пуст: том homeapp ещё не смонтирован.
  2. При старте пода init-контейнер выполняет init-dev-env.sh: на уже смонтированном томе homeapp создаёт нужные каталоги (mkdir -p /home/app/homeapp/.X) и передаёт их пользователю app.

После этого любая запись по пути ~/.X проходит через симлинк на PVC homeapp. Клон монорепо init-скрипт делает однократно: каталог /workspace заполняется, только если в нём нет маркера .gitclone. По завершении клона маркер ставится, и повторные старты пода клон не запускают.

Кеши по компонентам

Симлинки задаёт Dockerfile.dev каждого компонента, а каталоги под них в томе создаёт init-скрипт. Что куда попадает:

Компонент Кеш Путь Том
app1 backend maven-репозиторий ~/.m2 homeapp
app1 backend сборка target/ (в дереве исходников) workspace
app1 frontend зависимости node_modules (в дереве исходников) workspace
app1 frontend кеш npm ~/.npm homeapp
app1 frontend прочий кеш инструментов ~/.cache homeapp
app2 backend кеш pip (скачанные wheel) ~/.cache homeapp
app2 backend пакеты pip install --user ~/.local homeapp
app2 frontend зависимости node_modules (в дереве исходников) workspace
app2 frontend кеш npm ~/.npm homeapp
app2 frontend прочий кеш инструментов ~/.cache homeapp

node_modules и артефакты сборки (target/, dist/) лежат внутри рабочей копии в /workspace, то есть на томе workspace без симлинков: dev-серверы фронтов ставят зависимости через npm install в каталог компонента (./node_modules), бекенд app1 кладёт target/ рядом с pom.xml. Кеш сборки Angular (.angular/) Angular CLI тоже держит в дереве проекта под /workspace, отдельного симлинка на homeapp для него нет.

~/.npm (фронты) и ~/.local (бек app2) вынесены на том homeapp отдельным изменением. До него эти каталоги оставались в эфемерном слое контейнера и пересоздавались при каждом рестарте, так что npm install и pip install --user каждый раз заново тянули пакеты из сети.

Жизненный цикл томов

PVC живут дольше пода. Они сохраняются при:

  • обновлении StatefulSet через converge (новый шаблон, новые переменные);
  • смене версии образа компонента (bump VERSION/CI_TAG);
  • перепланировке пода на другую ноду и перезагрузке ВМ или ноды.

Тома уничтожаются только при явной очистке окружения – сносе релиза с удалением неймспейса и его PVC (см. Операции kube_ci, раздел про dismiss). reclaimPolicy standard-класса – Delete: PVC удаляется вместе со своим claim’ом, но не при рестарте пода, так что переустановка пода кеши не затрагивает.

На практике первый клон, первый npm install, первый pip install --user и первая сборка – разовая плата. Последующие операции в том же окружении идут из кеша на томах и от доступности сети не зависят.

Проверка

Симлинки кешей указывают внутрь смонтированных томов. Проверка из подключённого пода (kubectl exec или терминал VS Code Remote):

# куда реально ведёт симлинк кеша -- результат должен быть под одним из томов
readlink -f ~/.npm        # ожидаемо: /home/app/homeapp/.npm
readlink -f ~/.m2         # ожидаемо: /home/app/homeapp/.m2  (бекенд app1)

# оба тома смонтированы как отдельные файловые системы
df -P /workspace /home/app/homeapp

Список и состояние PVC окружения – снаружи кластера:

kubectl --context <ctx> -n <namespace> get pvc

Каждый под публикует пару workspace-<под> / homeapp-<под> в статусе Bound.
