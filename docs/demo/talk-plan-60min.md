# План доклада на 60 минут

> **Доставка по контракту: два стека -- один `werf converge`**
> Разработка прямо в dev-кластере, где dev неотличим от prod.
> Дистилят практики Kubernetes 2020-2025.

Раскладка доклада, который показывает связку `kube_ci` с двумя продуктами на
аудитории девопсов и разработчиков среднего уровня. План делит выступление на
одиннадцать блоков с таймингом; сумма блоков -- около 52 минут материала и 8
минут вопросов. Для каждого блока указаны цель, ключевые тезисы, что показывать
на экране (статья, диаграмма или команда) и какая статья документации служит
материалом для углубления. Доклад ведёт инженер, который сам разворачивает демо.

Тяжёлые операции -- запуск dev-серверов в подах и сборку через converge -- лучше
прогнать заранее или параллельно на машине деплоя, а на экране держать готовый
результат, чтобы выкатка не съедала время. Это особенно важно для блока с живой
демонстрацией цикла: dev-серверы поднимают до доклада, на сцене остаётся только
правка и мгновенный отклик.

Доклад построен как дистилляция личного опыта разработки приложений под
Kubernetes за 2020-2025 годы. Контур и приёмы выросли из реальной работы, а не
из учебного примера, и это задаёт тон всему выступлению.

Материал ранжирован по важности. Ядро часа -- три темы, на них приходится
максимум времени и акцента:

1. развязка контрактом `.helm/def.sh` плюс `werf converge`; два контрастных
   продукта служат доказательством, что в контуре нет знания о приложении;
2. разработка прямо в кластере и живой цикл работы: dev почти идентичен prod,
   отлаживается сама логика Kubernetes (Ingress, пробы, service discovery), а
   цикл правка -> dev-сервер -> отладка -> тест -> push -> converge показывается
   вживую -- это пик доклада;
3. операции доставки на живых командах: converge, rollback, dismiss, purge, dev
   и prod одним контуром.

Остальное сжато в контекст: компромиссы схемы, выход во внешний CI, метрики
DORA, секреты и версионирование упоминаются с отсылкой к профильным статьям, без
углубления.

## Блок 2. Проблема доставки и werf converge (6 мин)

Цель -- объяснить, почему доставка в Kubernetes распадается на две половины, и
ввести `werf converge` как инструмент, который их сшивает из одного git-состояния.

Тезисы:

- доставка делится на сборку образа и применение релиза; большинство
  инструментов закрывают только одну половину, а стык (синхронизация тегов,
  пересборка изменившихся слоёв) держит инженер скриптами;
- эта работа повторяется в каждой команде и плохо переносится между проектами;
- werf совмещает сборку и применение релиза в одной программе и одном запуске;
- центральная команда -- `werf converge`: собирает недостающие образы, публикует
  их в registry и применяет релиз; идемпотентна, в демо запускается атомарно;
- сборка разбита на кешируемые стадии, werf пересобирает только затронутое
  правкой;
- по умолчанию werf требует git (giterminism); в демо это требование ослаблено
  ради сборки из рабочего дерева -- к этому вернёмся в блоке про контекст.

Показать: схему «сборка -> registry -> релиз»
([industrial-cicd.png](../pics/industrial-cicd.png)); [поток
converge](../pics/converge-flow.png); строку активации
`source "$(~/bin/trdl use werf 2 stable)"`.

Материал для углубления: [доставка в Kubernetes](../concepts/delivery-to-k8s.md),

## Блок 3. Контур kube_ci и контракт `.helm/def.sh` (6 мин)

Цель -- показать, что `kube_ci` -- тонкая bash-обёртка над werf, а развязка с
продуктом держится на контракте `.helm/def.sh`. Это первая тема ядра.

Тезисы:

- `kube_ci` сведён к базовым операциям над уже развёрнутым кластером; логика
  вынесена в общие библиотеки `utils/`, окружения подмешивают локальные
  `k8s_defs` и `productlist`;
- контур не знает внутренностей продукта -- он работает по контракту
  `.helm/def.sh`: env-функция окружения экспортирует `APPNAME`, `ENVNAME`,
  `CI_URL`, `CI_TAG` и опционально `NAMESPACE`;
- `deploy()` читает только реально экспортированные `CI_*`-переменные (`${!CI_@}`)
  и пробрасывает их в helm как `--set`; `CI_TAG` идёт отдельным тегом образов;
- весь обмен между продуктом и контуром идёт через эти переменные и набор образов
  из `werf.yaml`; подключить новый продукт можно без правок в `kube_ci`.

Показать: [диаграмму архитектуры](../pics/architecture.png) (связка apps ->
kube_ci -> кластер); [контракт приложения](../concepts/application-contract.md),
блок с примером `def.sh`.

## Блок 5. Ядро: разработка в кластере, dev≈prod, отладка k8s (9 мин)

Цель -- показать главный приём схемы: разработка идёт прямо в кластере, dev почти
идентичен prod, поэтому отлаживается сама логика Kubernetes, а не одни
контейнеры. Это вторая тема ядра и подводка к живой демонстрации.

Тезисы:

- в dev продукт разворачивается не Deployment'ом, а долгоживущим
  подом-песочницей (`sleep infinity`) на базе StatefulSet, к которому разработчик
  подключается через VS Code Remote и пишет код прямо в поде;
- dev собран из тех же кубовых объектов, что prod: чарт, Ingress, Service,
  StatefulSet с Postgres, ConfigMap, Secret -- класс расхождений «локально
  работало, в кластере нет» почти исчезает;
- в кластере отлаживается то, чего docker-compose не показывает:
  Ingress-маршрутизация (`/api`, `/pgadmin`, host по nip.io, rewrite), service
  discovery по DNS и `targetPort`, readiness/liveness-пробы, поведение rollout и
  `updateStrategy: OnDelete` у Postgres, монтирование Secret и ConfigMap,
  init-контейнеры, сам `werf converge` с хуками;
- docker-compose моделирует только контейнеры и docker-сеть -- Kubernetes-слоя в
  нём нет вовсе, поэтому это другая модель, а не упрощённая копия dev;
- один артефакт доставки без дрейфа: dev и prod различаются только окружением в
  `.helm/def.sh`; compose стал бы вторым параллельным артефактом и двойной
  поддержкой;
- тяжёлое (maven, node, pip, кеши, базы) живёт на ноде кластера, VS Code Remote
  остаётся лёгким редактором и не нагружает ноутбук.

Показать: [разработку внутри кластера](../delivery/dev-in-cluster.md), раздел
«Зачем dev в кластере, а не docker-compose»; список того, что отлаживается в
кластере и не видно в compose; вывод `kubectl -n app1-java-react-dev get pods` с
поднятым dev-подом.

Материал для углубления: [разработка внутри

## Блок 6. Ядро: живой цикл работы на app1 (демонстрация, 10 мин)

Цель -- показать вживую полный цикл работы в dev-поде на фронте app1: правка
компонента -> мгновенный Vite HMR в браузере через ingress -> git push ->
converge в dev с новой версией. Это пик доклада.

Тезисы:

- цикл делится на две зоны: внутренняя петля (правка -> отладка -> тест)
  замыкается на dev-поде и повторяется сколько угодно раз; внешний контур (push,
  выкат через `kube_ci`) проходится реже;
- синхронизация между зонами -- только через git: ни `scp`, ни `rsync` между
  рабочей машиной и деплой-сервером;
- внутри пода dev-сервер фронта app1 поднимается скриптом
  [frontend/dev-start.sh](../../apps/app1-java-react/frontend/dev-start.sh)
  (`npm install` + `vite --host 0.0.0.0 --port 8080`); сервер виден снаружи через
  ingress по nip.io, потому что слушает `targetPort` dev-Service и в
  [vite.config.ts](../../apps/app1-java-react/frontend/vite.config.ts) задан
  `allowedHosts` (`.nip.io`);
- правка React-компонента подхватывается Vite HMR мгновенно -- браузер обновляет
  кадр без перезагрузки, и это видно зрителю;
- после проверки правка уходит `git push` с рабочей машины, затем converge в dev
  выкатывает релиз с новой версией.

Мини-сценарий демонстрации (dev-сервер поднят заранее, браузер открыт на хосте
ingress):

1. В подключённом к поду VS Code Remote открыть рабочую копию фронта app1 в
   `/workspace/werf_ci_demo/apps/app1-java-react/frontend`.
2. Выполнить `git pull` в `/workspace/werf_ci_demo`, чтобы рабочая копия была
   актуальна.
3. Править видимый React-компонент (заголовок, текст, цвет) и сохранить файл.
4. Показать браузер на `app1-java-react-dev-<NODE_IP>.nip.io`: Vite HMR обновляет
   кадр мгновенно, без перезагрузки страницы.
5. Сделать `git push` правки с рабочей машины.
6. Показать заранее прогнанный (или запущенный параллельно) converge в dev:
   `./00-build-deploy.sh app1-java-react` поднимает релиз с новой версией
   `CI_TAG`.

Запасной вариант: если HMR или ingress не отвечает на сцене, показать заранее
снятый результат -- скриншот или короткую запись правки и обновления кадра, плюс
готовый вывод converge. Сборку в любом случае держать прогнанной заранее, чтобы
она не съедала время блока.

Показать: диаграмму [цикла работы в dev-поде](../pics/dev-workflow-cycle.png);
живую правку фронта app1 и отклик в браузере; вывод converge с новой версией.

Материал для углубления: [типовой цикл работы в

[разработка внутри кластера](../delivery/dev-in-cluster.md),

## Блок 7. Кеши и persistent-volume'ы dev-подов (3 мин)

Цель -- кратко закрыть вопрос, почему повторный вход в петлю не платит за
пересоздание состояния пода.

Тезисы:

- dev-StatefulSet объявляет два PVC на под: `workspace` (`/workspace` -- клон
  монорепо, исходники, `node_modules`, артефакты сборки) и `homeapp`
  (`/home/app/homeapp` -- кеши пакетных менеджеров и `.vscode-server`);
- симлинки из домашнего каталога (`~/.m2`, `~/.npm`, `~/.cache`, `~/.local`)
  ведут на том `homeapp`, поэтому кеш пакетов и расширения VS Code переживают
  пересоздание пода и перезагрузку ВМ;
- первый клон, первый `npm install` и первая сборка -- разовая плата; дальше
  под собирается из локального кеша без обращения к сети;
- тома уничтожаются только при сносе релиза с удалением неймспейса, а не при
  рестарте пода.

Показать: [кеши и persistent-volume'ы dev-сред](../delivery/dev-caches-and-volumes.md),

## Блок 8. Ядро: операции на живых командах, dev/prod (6 мин)

Цель -- провести зрителя по жизненному циклу доставки на живых командах и
показать, что dev и prod обслуживаются одним контуром. Это третья тема ядра.

Тезисы:

- публикация -- `./pull_products.sh && ./00-build-deploy.sh [--all|<product>]`:
  pull связывает `apps/<product>` symlink'ом в `products/`, build-deploy вызывает
  converge на каждый выбранный продукт;
- откат версии -- `./03-rollback.sh <product> [revision]`: без revision печатает
  `helm history` (список выкаченных версий), с revision делает `helm rollback`;
  это ЧП-возврат к рабочей ревизии за секунды -- образ уже в registry, без
  пересборки; БД откат не трогает;
- снос -- `./01-dismiss.sh <product>|--all`: удаляет релиз вместе с неймспейсом,
  требует явного аргумента и без него отказывается работать;
- очистка -- `./02-purge-stages.sh`: сбрасывает локальный кеш стадий сборки, не
  трогает развёрнутые релизы;
- каждое окружение -- самодостаточный каталог `kube_ci/dev/` и `kube_ci/prod/` с
  одинаковым набором файлов; релиз уходит в неймспейс `<NAMESPACE>-<ENVNAME>`,
  поэтому одно приложение в dev и prod не конфликтует на одном кластере;
- временно оба окружения указывают на один физический preprod-кластер; при
  переходе на раздельные кластеры меняется только `KUBECONTEXT` и `REGISTRY` в
  `k8s_defs`.

[runbook деплоя](../runbooks/deploy.md); живой прогон публикации одного продукта
в dev, затем `kubectl -n <app>-dev get pods`; `./03-rollback.sh <product>` с
историей ревизий; вывод `kubectl get ns` с неймспейсами обоих окружений.

Материал для углубления: [операции kube_ci](../delivery/kube-ci-operations.md),

## Блок 9. Контекст: компромиссы, CI, DORA, секреты, версии (2 мин)

Цель -- честно назвать послабления демо и обозначить, куда схема растёт, без
углубления. Перечисление с отсылками к профильным статьям.

Тезисы:

- послабления демо обратимы: insecure-registry (HTTP / skip-TLS) ради
  in-cluster registry на nip.io, `--loose-giterminism` (сборка из рабочего
  дерева), общий preprod-кластер для dev и prod, хосты на nip.io вместо реального
  DNS и TLS, ключ `WERF_SECRET_KEY` на машине сборки, dev-под с примонтированным
  ssh-ключом -- при переносе в прод меняется конфигурация окружения, не логика
  доставки;
- те же скрипты оборачиваются в пайплайн внешнего CI: публикация --
  `deploy-dev`/`deploy-prod`, откат -- `rollback`, очистка -- `cleanup`; типовой
  поток -- автодеплой в dev, ручной промоут в prod;
- четыре метрики DORA снимаются с артефактов контура (статусы и метки времени
  job-ов, git-история с `VERSION`, факты запуска скриптов); кода сбора в
  репозитории нет, описана концепция;
- секреты: пароли в `.helm/secrets-<env>.yaml` шифрует werf по `WERF_SECRET_KEY`;
  версия едина на продукт -- `VERSION` раскатывается `set-version.sh` и отдаётся
  как `CI_TAG`.

Показать: [компромиссы и безопасность
схемы](../concepts/security-and-tradeoffs.md) одним кадром-перечнем.

## Блок 10. Выводы (2 мин)

Цель -- собрать доклад в одно утверждение.

Тезисы:

- контур опирается на контракт `.helm/def.sh`, поэтому два разнородных продукта
  доставляются одним кодом;
- разработка идёт прямо в кластере: dev почти идентичен prod, отлаживается сама
  логика Kubernetes, а цикл правка -> push -> converge замыкается через git;
- werf сшивает сборку и релиз из одного git-состояния и сводит доставку к
  набору операций;
- послабления демо понятны и обратимы -- при переносе в прод меняется
  конфигурация окружения, не логика доставки;
- всё это -- дистилляция практики 2020-2025, а не учебная конструкция.

Показать: [диаграмму архитектуры](../pics/architecture.png) ещё раз как финальный
кадр.
